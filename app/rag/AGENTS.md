# AGENTS.md

## Scope
- 本目录负责 RAG 的配置、索引、召回、融合、重排、上下文构建。
- 本目录的目标是为合同审阅提供“外部法律依据 + 内部审阅规则”的可追溯检索增强。
- 未经明确要求，不要修改 `app/router/`、`app/models/`、审阅结果 schema、主审阅接口返回结构。

## Architecture Rules
- 必须将知识库拆分为两个 collection：
- `external_legal_kb`
- `internal_review_rules`
- 禁止将内部规则库与外部法律库混存到同一个 collection。
- Qdrant 必须作为独立容器部署；禁止替换为内嵌式向量库或临时本地文件索引。
- 本地模型推理仅允许轻量 embedding 模型；未经明确批准，禁止引入重型本地 reranker 或大模型本地推理。

## Retrieval Rules
- 必须使用混合检索；禁止只保留 dense 或只保留 sparse 检索。
- 首轮召回必须至少同时包含 semantic 信号与 lexical 信号。
- 必须使用多路召回提升 recall；禁止仅用单一 query 直接检索。
- 每个 collection 的多路召回结果必须显式融合；禁止简单拼接后直接截断。
- 融合策略、权重、top_k、过滤条件的改动默认视为高风险改动。
- 重排只允许作用于候选集；禁止对全库做 rerank。

## Knowledge Base Rules
- 外部法律库必须覆盖：法律、司法解释、行业监管规范、地方法规。
- 内部规则库必须只存内部审阅规则、风险规则、标签规则、审阅口径。
- 法律文本切分优先按“法/章/节/条/款/项”结构，不要先按固定长度粗切。
- 每条知识记录必须带完整元数据；缺失关键元数据的文本禁止入库。
- 失效、废止、来源不明、状态不清的法规默认不得进入生产召回。
- 地方法规必须标记适用地区；行业监管规范必须标记适用行业。
- 内部规则必须标记启用状态与适用范围；禁用规则不得参与线上召回。

## Provider Rules
- Embedding 必须同时支持 `local` 和 `remote` 两种 provider 模式。
- provider 切换必须通过配置实现；禁止在业务代码中硬编码切换逻辑。
- 本地 provider 默认只允许轻量 embedding 模型。
- 在线 provider 必须走统一客户端封装；禁止在检索逻辑中直接散落第三方 SDK 调用。
- 修改 embedding 模型或向量维度时，必须明确说明是否需要重建索引。

## Prompt Injection Rules
- 拼入审阅模型的 RAG 内容必须使用受控模板；禁止把原始检索结果直接拼进 prompt。
- 注入上下文必须保留来源信息，至少包括标题、条号或规则编号、来源类型。
- 必须限制注入条数与 token 预算；禁止无限制拼接召回文本。
- 外部法律依据与内部审阅规则必须分段展示，不能混写为单段文本。
- 当召回为空时，必须有明确降级路径，不能阻断主审阅流程。

## Filtering And Compliance
- 外部法律检索必须支持按地区、行业、效力状态过滤。
- 内部规则检索必须支持按合同类型、风险标签、组织范围过滤。
- 除非任务明确要求，禁止放宽默认过滤条件。
- 任何影响法规适用范围判断的改动，都必须说明是否会导致误召回。

## Validation
- 至少验证以下场景：
- 命中正确法规
- 命中正确内部规则
- 空召回
- 多路结果重复融合
- 地区过滤
- 行业过滤
- 失效法规过滤
- 内部规则禁用过滤
- 评估优先看结构指标与样例集，不要只凭主观案例声称“效果提升”。
- 没有离线评估集时，禁止声称召回质量已被验证。

## Safety
- 禁止在日志中打印完整法规正文、内部规则全文、用户合同原文。
- 调试输出只允许打印 ID、标题、条号、分数、命中来源。
- 禁止把密钥、远程 embedding 配置、Qdrant 凭据写死在代码中。

## Change Reporting
- 每次改动必须说明：
- 影响的 collection
- 影响的 provider
- 影响的召回链路
- 是否需要重建索引
- 是否影响 prompt 注入格式
- 验证范围与未验证风险
