生成时间：2026-04-06

## 评估对象

本次对比基于 `result/` 目录下三份最新结果文件：

1. `linear_pipeline_重庆第二师范学院南山校区学生宿舍三期建设工程EPC总承包合同2025.4.27最终版_20260406_173536.txt`
2. `merge_arbitration_重庆第二师范学院南山校区学生宿舍三期建设工程EPC总承包合同2025.4.27最终版_20260406_180259.txt`
3. `hierarchical_evidence_checker_重庆第二师范学院南山校区学生宿舍三期建设工程EPC总承包合同2025.4.27最终版_20260406_194113.txt`

说明：

- 本文评分是工程侧评估，不是基于人工标注答案集的严格准确率。
- “完整度”主要看覆盖范围、风险类型广度、跨条款问题是否能被纳入。
- “精确度”主要看结论是否具体、证据是否贴合、是否存在占位语句、重复或泛化输出。
- “运行效率”只结合本次真实运行时间，不代表线上最终性能。

## 当前 3 种 agent 架构链路

### 1. 线性顺序流水线

链路：

`解析合同 -> 按长度分块 -> 每个 chunk 独立审阅 -> 汇总所有修改建议 -> 输出结果`

特点：

- 结构最简单。
- 每个分块基本独立，没有显式全局冲突处理。
- 输出量大，适合快速得到“问题池”，但容易重复。

### 2. 归并层仲裁

链路：

`解析合同 -> 按长度分块 -> chunk agent 并发结构化分析 -> 全局意见池 -> 冲突检测 -> 主 agent 仲裁 -> 输出最终仲裁意见`

特点：

- 比线性流水线多了全局冲突检测和归并。
- 能处理字段冲突、建议冲突、跨块联动。
- 输出结果更收束，运行效率也更好。

### 3. 层级主从 + 共享证据层 + checker

链路：

`确定性预处理 -> 构建共享证据快照 -> planner 派发 specialist 任务 -> 依赖解析生成 context package -> specialist 并发审阅 -> merge 归并 -> checker 校验 -> 输出最终结论`

特点：

- 结构最完整，最接近后续可扩展的多 agent 方向。
- 已经引入“证据层、任务层、checker 层”。
- 对复杂合同更有扩展潜力，但当前结果仍偏多，成本最高。

## 分项分析

### 1. 线性顺序流水线

优点：

- 覆盖面较广，184 条建议说明扫描强度足够。
- 单块内问题表达通常较具体，很多建议改写文本很完整。
- 适合做早期问题发现和 prompt 试错。

不足：

- 没有跨块归并，同类问题和前后条款冲突会重复出现。
- 输出极长，阅读成本高。
- 结果中存在 `评分60` 这类噪音字段，说明结果口径不够干净。
- 有少量 `未找到修改建议` 之类的弱结果，稳定性一般。

评价：

- 完整度高于归并 demo，因为它几乎把每块都“展开说了一遍”。
- 精确度不算高，主要问题不是单条结论太差，而是重复、冗长、全局一致性不足。
- 运行时间接近 487 秒，但没有换来更好的收束性，性价比一般。

### 2. 归并层仲裁

优点：

- 三者里整体最均衡。
- 能显式处理字段冲突、建议冲突和跨块联动。
- 运行时间最短，约 262 秒，明显优于另外两种。
- 最终结果体量最小，便于阅读和对比。

不足：

- 为了压缩结果，牺牲了一部分细节覆盖。
- 少数最终结论仍然带有占位痕迹，如 `本块原文中的问题`、`修改建议`。
- 一些冲突项的证据仍偏模板化，例如“多个分块提取的关键字段值不一致”“部分修改建议会直接影响其他分块中的对应字段”，解释力还不够强。

评价：

- 完整度略低于另外两种，因为它主动压缩了输出。
- 精确度在三者中最好，不是因为每条都最强，而是因为整体噪音最低、结构最收束、全局视角最好。
- 如果当前阶段目标是“做一个可读、可比较、可继续迭代的默认 demo”，这一版最适合作为基线。

### 3. 层级主从 + 共享证据层 + checker

优点：

- 架构最先进，最接近后续真正多 agent 化的方向。
- 已具备 planner、specialist、evidence、checker 这几个关键层次。
- 单条结论普遍更像“主题化审阅意见”，不少问题说明和建议比较深入。
- 已显式记录证据锚点、依赖句柄、来源 specialist，审计性优于前两种。

不足：

- 结果过多，234 条最终发现基本没有真正收束，说明当前 merge 去重力度不足。
- checker 全部接受，`需复核数量=0`，这在工程上通常意味着 checker 门槛偏宽，暂时没有真正发挥“质检拦截”作用。
- 同一问题会从不同 specialist 视角重复表达，例如同一条低价风险担保条款被多个 specialist 分别提出相近结论。
- 运行时间最长，约 560 秒，成本明显偏高。

评价：

- 完整度三者最高，因为它不仅扫当前分块，还引入了定义、依赖和多 specialist 视角。
- 精确度高于线性流水线，但低于归并仲裁。主要不是单条不准，而是“重复多、checker 偏松、结果未收束”。
- 当前更像“架构正确但参数与后处理尚未调顺”的版本。

## 结论

### 综合排序

1. 归并层仲裁
2. 层级主从 + 共享证据层 + checker
3. 线性顺序流水线

### 结论解释

- 如果看“当前 demo 阶段的综合可用性”，归并层仲裁最好。
  原因是它在运行时间、结果体量、全局一致性三项之间取得了最好的平衡。
- 如果看“后续演进潜力”，层级主从 + 共享证据层 + checker 最强。
  原因是它已经具备后续接入真实 RAG、强化 checker、增加 specialist、做多轮依赖展开的天然结构。
- 线性顺序流水线更像基准版和问题收集器，适合做底层 prompt 对照，不适合直接作为默认多 agent 方案。

## 后续建议

### 对线性顺序流水线

- 保留为 baseline。
- 不建议继续往这个范式上叠复杂逻辑。

### 对归并层仲裁

- 建议继续作为当前默认展示 demo。
- 优先修复少量占位输出。
- 继续加强冲突证据描述，不要只给模板化结论。

### 对层级主从 + 证据层 + checker

- 下一步优先做两件事：
- 第一，压缩重复 findings，减少多 specialist 对同一问题的平行重复输出。
- 第二，增强 checker 拦截规则，让一部分弱证据、弱依赖、弱建议进入 `needs_recheck`，否则 checker 形同“全量放行”。

## 最终建议

当前阶段建议：

- 默认对外展示：`归并层仲裁 demo`
- 继续研发主线：`层级主从 + 共享证据层 + checker demo`
- 保留基线对照：`线性顺序流水线 demo`

这样分工最清晰：

- 线性版负责做最原始的覆盖能力对照。
- 归并版负责做当前最平衡的演示方案。
- 层级证据版负责做未来主方案的架构试验田。
