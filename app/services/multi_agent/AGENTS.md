# AGENTS.md

## Scope
- 本文件仅适用于 `app/services/multi_agent/` 目录。
- 本目录用于存放不同设计范式的多 agent 实现 demo、实验脚本、对比样例与配套文档。
- 本目录默认定位为实验区；未经明确要求，不要将这里的 demo 直接接入主审阅接口、主任务编排链路或线上入口。

## Goals
- 本目录允许先沉淀通用工具层，再基于工具层扩展具体设计范式 demo。
- `config.py` 必须维护本目录独立配置，至少包括 demo 模型配置、测试合同目录路径、结果输出目录。
- 每个 demo 必须明确说明其设计范式，例如：Supervisor、Planner-Executor、Reviewer-Refiner、Router、Debate、Workflow Graph。
- 每个 demo 必须优先体现结构、职责划分、消息流转、失败兜底，不要把重点放在 prompt 堆砌上。
- 新增 demo 时，优先保证“可读、可跑、可替换”为目标，避免一开始就做成强耦合实现。

## Change Boundaries
- 不要借实现 demo 的名义重构 `app/services/` 现有生产链路。
- 未经明确要求，不要修改 `app/router/`、`app/models/`、数据库结构、鉴权、中间件、部署脚本。
- 未经明确要求，不要改动现有合同审阅主流程中的模型参数、输出结构、缓存键、并发策略、落库顺序。
- 若 demo 需要复用现有服务能力，优先采用薄封装或适配层；不要直接侵入式改写既有模块。

## Demo Rules
- 若存在目录级通用工具文件，其职责应只限于解析、分块、单步审阅、共享数据结构、配置读取等基础能力；不要在工具文件中提前固化某一种多 agent 编排范式。
- 每个 demo 必须尽量独立，避免多个 demo 之间共享隐式状态。
- 每个 demo 必须有清晰入口，文件命名应能直接看出设计范式与用途。
- 每个 demo 的运行结果必须落到 `app/services/multi_agent/result/` 下；至少记录审阅完整结果文本和总计耗时。
- 多 agent 之间的消息、状态、上下文对象必须使用明确结构；禁止在多个步骤之间传递随意拼接的自由文本字典。
- 调度逻辑、角色定义、工具调用、输出整理应尽量分层，避免所有逻辑堆在单文件单函数中。
- 若引入新的抽象层，必须先说明它解决的是哪类多 agent 复用问题，避免为抽象而抽象。

## LLM And Prompt Rules
- 本目录中的 demo 可以探索不同协作范式，但默认不要修改仓库其他目录中的生产 prompt。
- 未经明确要求，不要随意改动生产使用的模型名称、temperature、top_p、max_tokens、SSE 输出结构。
- demo 内若使用 prompt 模板，必须保持输出字段可解析，并为解析失败提供兜底。
- demo 内若模拟多角色协作，必须明确每个角色的输入、输出、停止条件与失败处理。

## Safety
- 禁止在 demo 中输出合同原文、OCR 全文、客户敏感信息、token、密钥。
- 调试日志应优先打印步骤名、角色名、状态、耗时、摘要，不要打印完整敏感文本。
- 未经明确要求，不要在 demo 中接入真实外部副作用能力，例如真实发送、真实写库、真实回调。

## Validation
- 每新增一个 demo，至少完成一次与该 demo 直接相关的最小结构验证。
- 验证优先关注：流程是否可走通、角色切换是否正确、状态传递是否完整、失败兜底是否生效。
- 若验证依赖外部模型、密钥、数据库、Redis 或其他不可用基础设施，必须提供本地可运行的最小替代验证方案，或明确说明未验证边界。

## Documentation
- 每新增一个 demo，必须补充该 demo 的设计说明，至少写清：
- 设计目标
- 角色分工
- 执行流程
- 输入输出
- 已知限制
- 若某个 demo 只是概念验证，必须明确标记其非生产可用状态。

## Change Reporting
- 每次在本目录下完成任务后，必须说明：
- 本次新增或修改了哪个 demo
- 是否影响现有生产链路
- 当前实现属于哪种多 agent 设计范式
- 已完成的最小验证
- 未验证风险与下一步建议
