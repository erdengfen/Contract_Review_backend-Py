# PLANS.md

本文档用于记录合同审阅后端的分步重构计划、开发进度、阶段验收和风险状态。后续涉及分步开发或长期重构任务时，必须同步更新本文档。

## 项目概览

- 项目定位：面向高校及企事业单位合同管理场景的 AI 合同审阅后端服务。
- 核心能力：合同文件上传与解析、合同审阅、合同比对、合同聊天、Prompt 管理、模型配置、看板统计、登录鉴权、RAG 检索和多 Agent 审阅实验链路。
- 技术栈：Python 3.11.x、FastAPI、Pydantic v2、SQLAlchemy、MySQL、Redis、Qdrant、SSE、OpenAI-compatible API、LangChain、LangGraph、Langfuse、uv。
- 当前重点：`app/services/` 中的合同审阅、文档处理、合同比对、多 Agent / LLM 调用链路。
- 当前约束：当前已存在的 FastAPI 接口入参字段与出参字段全部冻结，重构不得修改字段名称、字段层级、字段含义和 SSE 输出结构。

## 重构总原则

- 先建立现状基线，再拆分实现，不在没有基线的情况下做大范围改动。
- 保持现有接口兼容，优先通过内部适配层、服务层拆分和测试补强降低风险。
- 不随意改动模型名称、模型参数、Prompt 输出格式、缓存键、并发控制、结果落库顺序和风险字段语义。
- 不做无关重命名、批量格式化、目录搬迁和跨模块重构。
- 每个阶段完成后执行最小验证；涉及接口和 AI 输出的阶段优先做结构验证。

## 已识别风险

- `docs/PROJECT_STRUCTURE.md` 中仍包含旧目录、旧启动命令和 `requirements.txt` 入口描述，后续需要与 uv、`pyproject.toml` 和当前目录结构对齐。

- LLM 与 Agent 链路涉及 Prompt、输出解析、模型参数、SSE、缓存键和落库顺序，属于高风险重构区域。
- 当前业务依赖 MySQL、Redis、Qdrant、OCR、外部 LLM 和可能的 MCP 服务，完整链路验收容易受环境影响，需要记录未验证边界。
